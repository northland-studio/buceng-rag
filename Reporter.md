## 一、开发概述

### 1.1 项目背景

本项目旨在构建一个基于RAG架构的Web应用，帮助研究人员分析Minecraft游戏内的社会学现象。系统采用本地向量知识库和云端大模型相结合的方式，既保证了数据安全，又提供了强大的分析能力。

### 1.2 开发周期

- 开始时间：2026-04-26
- 完成时间：2026-04-26
- 总耗时：1天
- 代码行数：约3000行

## 二、技术实现

### 2.1 架构设计

系统采用分层架构：

```
┌─────────────────────────────────────┐
│         表现层 (Streamlit)          │
├─────────────────────────────────────┤
│         业务逻辑层                   │
│  ┌──────────┬──────────┬──────────┐ │
│  │ 嵌入模型 │ 知识库   │ LLM API  │ │
│  └──────────┴──────────┴──────────┘ │
├─────────────────────────────────────┤
│         数据层                       │
│  ┌──────────┬──────────┐            │
│  │ ChromaDB │ 文件系统 │            │
│  └──────────┴──────────┘            │
└─────────────────────────────────────┘
```

### 2.2 核心模块

#### 2.2.1 配置管理（config.py）

**功能**：
- 使用pydantic-settings进行配置管理
- 支持环境变量和.env文件
- 提供类型验证和默认值

**关键决策**：
- 选择pydantic而非手动解析，提高代码可维护性
- 支持配置验证，避免运行时错误

#### 2.2.2 嵌入模型（embedding.py）

**功能**：
- 加载本地嵌入模型（BAAI/bge-large-zh-v1.5）
- 提供单文本和批量嵌入功能
- 实现模型缓存和单例模式

**关键决策**：
- 使用单例模式避免重复加载模型
- 支持CPU和CUDA设备切换
- 添加输入验证和错误处理

#### 2.2.3 知识库（knowledge_base.py）

**功能**：
- 管理ChromaDB向量数据库
- 实现理论卡片的增删改查
- 支持向量检索和相似度计算
- 管理黄金样本数据

**关键决策**：
- 使用ChromaDB而非其他向量数据库，因其轻量且易用
- 实现自定义嵌入函数，与本地模型集成
- 支持自动加载初始数据

#### 2.2.4 LLM API（llm_api.py）

**功能**：
- 调用DeepSeek API生成分析
- 支持流式和非流式输出
- 实现重试机制和错误处理

**关键决策**：
- 使用OpenAI SDK而非直接HTTP请求，提高兼容性
- 实现指数退避重试策略
- 支持流式输出提升用户体验

#### 2.2.5 工具函数（utils.py）

**功能**：
- 输入验证和清洗
- 引用解析和高亮
- 文件操作辅助

**关键决策**：
- 提供统一的输入验证接口
- 支持多种引用格式解析

## 四、重大决策

### 4.1 技术选型决策

#### 决策1：选择ChromaDB作为向量数据库

**背景**：需要选择一个轻量级、易用的向量数据库。

**选项**：
1. ChromaDB - 轻量级，支持本地模式
2. Pinecone - 云端服务，需要网络
3. Milvus - 功能强大，但部署复杂

**决策**：选择ChromaDB

**理由**：
- 支持本地持久化，符合隐私要求
- 安装简单，无需额外依赖
- API友好，易于集成

#### 决策2：使用Streamlit作为Web框架

**背景**：需要快速构建数据科学应用的Web界面。

**选项**：
1. Streamlit - 快速原型，适合数据应用
2. Flask - 灵活，但需要更多代码
3. FastAPI - 性能好，但前后端分离

**决策**：选择Streamlit

**理由**：
- 开发速度快，适合原型开发
- 内置组件丰富，无需前端知识
- 支持缓存和会话状态

#### 决策3：使用pydantic进行配置管理

**背景**：需要管理大量配置项，并进行验证。

**选项**：
1. pydantic - 类型安全，自动验证
2. python-dotenv - 简单，但无验证
3. configparser - 标准库，但功能有限

**决策**：选择pydantic

**理由**：
- 类型安全，IDE支持好
- 自动验证，避免配置错误
- 支持嵌套配置和复杂类型

### 4.2 架构决策

#### 决策4：采用单例模式管理模型实例

**背景**：嵌入模型和知识库需要全局唯一实例。

**决策**：使用单例模式

**理由**：
- 避免重复加载模型，节省内存
- 确保知识库状态一致
- 简化资源管理

#### 决策5：实现自定义异常体系

**背景**：需要统一的错误处理机制。

**决策**：创建自定义异常类

**理由**：
- 提供清晰的错误信息
- 支持错误分类和处理
- 便于日志记录和调试
